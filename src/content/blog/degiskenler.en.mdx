[In the previous post](/en/blog/pseudocode), while writing pseudocode, we used lines like
`n ← 1` or `total ← 0`. But we never stopped to ask: **what exactly is this `n`, this
`total`?** These little boxes we put values into have a name: **variables.**

In this post, without writing any code, we'll understand what a variable is from the very
beginning, with everyday examples. Because a variable is how a program **remembers
information** — and it sits at the heart of nearly every algorithm you'll write.

<Callout type="note" title="Where are we in this series?">
This is the fourth post in the **Algorithms** series. So far we got to know the algorithm,
drew it with a flowchart and wrote it in pseudocode. Now we step *inside* algorithms to
look at their first building block: the boxes that store information, i.e. variables.
</Callout>

## A memory problem

Think of it like this: you send a friend shopping and tell them over the phone. "Buy
three loaves of bread," you said. Your friend has to **keep that in mind**, right? They
make a mental note: "bread = 3".

A computer does the same. To use a piece of information later, it has to **put** it
somewhere and give that place a **name**. That place is the variable.

> A **variable** is a box we put a piece of information into, with a **name (a label)** on
> it. When you say the box's name, the computer brings you the value inside.

## The labeled box

Keep this picture in mind: a box with a label on it.

- The box's **name** (label): `age`
- The **value** inside: `25`

You can draw it on paper like this:

```text title="A variable" showLineNumbers=false
┌───────────┐
│   age     │   ← the box's name (label)
│    25     │   ← the value inside
└───────────┘
```

Now when you say `age`, the computer understands "25". Once you've set the name, you don't
have to memorise the value; you just use the name.

<Callout type="note" title="Where are these boxes, physically?">
These boxes really exist: they are kept in the computer's **memory (RAM)**. Think of memory
as a giant cabinet with millions of tiny slots. When you say `age ← 25`, the computer finds
an empty slot, puts 25 inside, and links that slot's location to the name `age`. You say the
name, it finds the right slot. You never deal with memorising addresses — that's the
convenience a variable gives you.
</Callout>

## Describing a person with boxes

A single box is nice, but the real power appears when you use several together. Say we want
to describe a user. One piece of information isn't enough; we use several boxes:

```text title="A user = several variables" showLineNumbers=false
name   ← "Mehmet"
age    ← 25
city   ← "Ankara"
active ← true
```

That's it: four boxes describing one person. We represent a game character (`health`,
`score`, `level`), a product (`name`, `price`, `inStock`) or an account the same way —
several variables used together. Each box holds one piece; together they form the whole.

## Putting something in the box: assignment

Putting a value into a variable is called **assignment**. In pseudocode we showed it with a
left arrow (`←`):

```text title="Assignment" showLineNumbers=false
age   ← 25            # put 25 into the "age" box
name  ← "Ada"         # write Ada into the "name" box
bread ← 3             # put 3 into the "bread" box
```

Don't read these left to right as "this into that", read them **right to left**: *first the
value 25 is prepared, then it is put into the `age` box.*

You can also put one variable's value into another. In that case the value is **copied** —
the source box stays as it is:

```text title="From variable to variable" showLineNumbers=false
a ← 7        # box a holds 7
b ← a        # a's value (7) is COPIED into b
a ← 99       # a changed but b is still 7; b got a copy of the value
```

<Callout type="important" title="Equals doesn't always mean 'is it equal?'">
Later, in real code, you'll often see assignment with an equals sign: `age = 25`. Here `=`
does **not ask** "is age equal to 25?" like in maths; it **commands** "put 25 into the age
box". This is the thing beginners confuse the most. To avoid confusion, we use the arrow
(`←`) in pseudocode.
</Callout>

## The life of a variable

Every variable lives a short life with three stages:

<Steps>

1. **Birth (creation):** You first give a name and put a value inside — `score ← 0`.
2. **Use (reading):** You say the name and read the value inside — `PRINT score`.
3. **Change (update):** You write a new value over it — `score ← score + 10`.

</Steps>

Trying to read a variable before it is born (before putting any value inside) is a typical
mistake — like trying to take something out of an empty box. Put first, then read.

## Doing operations with variables

Variables don't just *store* information; their real power is that you can do **operations**
on them. You do arithmetic with numbers:

```text title="Operations with numbers" showLineNumbers=false
width  ← 5
height ← 3
area   ← width * height      # area = 15
```

`+` add, `-` subtract, `*` multiply, `/` divide. Remember from last time: `MOD` gives the
remainder of a division (`7 MOD 3 = 1`). You can combine several variables in one formula —
for example, let's convert Celsius to Fahrenheit:

```text title="Temperature conversion" showLineNumbers=false
celsius    ← 25
fahrenheit ← celsius * 9 / 5 + 32    # 77
```

You can **join** texts together (this is called *concatenation*); usually `+` is used again:

```text title="Joining texts" showLineNumbers=false
first ← "Ada"
last  ← "Lovelace"
full  ← first + " " + last    # full = "Ada Lovelace"
```

Note: without the `" "` (a quoted space) in the middle, the result would be `"AdaLovelace"`.
The computer does exactly what you say — you even have to put the space yourself.

## Why "variable"? Because its value varies

That's exactly where the name comes from: a variable's value **can change.** If you put a
new value into the same box, the old value is lost:

```text title="The value changes" showLineNumbers=false
score ← 10         # the score box now holds 10
score ← 25         # the old 10 is erased; score now holds 25
score ← score + 5  # score = 25 + 5 = 30
```

Look at the last line: `score ← score + 5`. This is not maths! It means "take the current
`score`, add 5, and put the result back into the `score` box". So the box is read first,
then a new value is written over it. This is exactly how you increase a counter.

### An old friend: swap

[In the first post](/en/blog/what-is-an-algorithm) we solved the "swap the contents of two
glasses" puzzle, remember? Now we see why that puzzle was so important: if you write over a
box without thinking, the old value is **lost.**

That's why swapping two variables' values needs a **temporary box**:

```text title="Swapping two values" showLineNumbers=false
temp ← a       # save a's value in the temp box
a    ← b       # put b's value into a
b    ← temp    # put the saved value into b
```

That "empty third glass" in the puzzle was exactly this `temp` variable.

## Following variables: the trace table

Because variables keep changing in an algorithm, the question "what's in which box right
now?" gets harder to follow. Programmers use a simple but powerful tool for this: the
**trace table.** Each row is a step, each column a variable; you "run" the algorithm on
paper, in your head.

Let's trace this little algorithm:

```text title="Steps to trace" showLineNumbers=false
counter ← 0
counter ← counter + 1
counter ← counter + 1
counter ← counter * 2
```

After each line, we write the value of `counter` into a table:

| Step | `counter` |
| ---- | :-------: |
| `counter ← 0`           | 0 |
| `counter ← counter + 1` | 1 |
| `counter ← counter + 1` | 2 |
| `counter ← counter * 2` | 4 |

At the end of the table we know **for certain** that `counter` is 4 — we didn't guess, we
traced it step by step. When you have a bug, this method is gold: you see exactly which step
went wrong.

## Computers' basic types: primitive types

What you put in a box matters, because a computer stores different **kinds** of information
differently and does different things with them. The most basic types, ones that can't be
broken into smaller pieces, are called **primitive types** — like the bricks every program
is built from. Don't let them look scary; they're all things you already know from everyday
life.

### Integer — things you count

Things that aren't split, counted one by one: the apples in a basket, your age, the number
of floors in a building, the students in a class. There's no such thing as "2.5 students".

```text showLineNumbers=false
age ← 25      quantity ← 3      floor ← 7
```

**Link to algorithms:** A counter in a loop is always an integer (`count from 1 to 5`); it
holds how many of something there are.

### Float — things you measure

Things that can be fractional, obtained by **measuring**: your height (1.78 m), your fever
(36.6°C), a product's price (9.99), the fuel in a tank (42.5 litres).

```text showLineNumbers=false
height ← 1.78      price ← 9.99      temperature ← 36.6
```

**Link to algorithms:** Averages, ratios, percentages and money calculations are usually
done with floats (`average ← total / 3`).

### Character (char) — a single symbol

A single letter, digit or sign; written in single quotes: `'A'`. Everyday counterparts: your
blood type (`A`, `B`, `0`), your grade (`A`, `F`), a `Y`/`N` answer to a question.

```text showLineNumbers=false
bloodType ← 'A'      grade ← 'B'      answer ← 'Y'
```

**Link to algorithms:** Used for "single-key choices" (a `Y`/`N` menu) or checking a single
letter.

### True / False (boolean) — two-state things

Information with only two values: is the light **on or off**, is the door locked, is the bill
paid, is it raining. Like an on/off switch.

```text showLineNumbers=false
doorOpen ← true      paid ← false
```

You usually **don't write** this type yourself — it appears on its own as the result of a
comparison:

```text title="Where true/false comes from" showLineNumbers=false
age   ← 20
adult ← age ≥ 18      # the adult box gets "true" (because 20 ≥ 18)
```

**Link to algorithms:** This is the fuel of all **conditions** (`IF … THEN`); the
[comparison operators](/en/blog/pseudocode) from last time (`≥`, `<`, `=` …) always produce
these true/false values. It's also used as a "flag" that marks a state.

### And: text (string) — a sequence of characters

A name, an address, a sentence... A **string** is actually made of **characters** placed
side by side: `"Ada"` = `'A'` + `'d'` + `'a'`. Written in double quotes.

```text showLineNumbers=false
name ← "Ada"      city ← "İstanbul"
```

**Link to algorithms:** Used to store a user's name, a message or an address.

Let's see them all together:

| Type | Everyday counterpart | Example | In algorithms |
| ---- | -------------------- | ------- | ------------- |
| Integer       | apple count, age      | `25`    | counter, quantity |
| Float         | height, temperature, price | `9.99` | average, ratio |
| Character (char) | blood type, grade  | `'A'`   | menu choice       |
| Boolean       | is the light on?      | `true`  | conditions, flag  |
| String        | name, address         | `"Ada"` | user input        |

<Callout type="note" title="What does 'primitive' mean, and why is text different?">
A **primitive** type is the most basic type that can't be broken into smaller pieces —
integer, float, character and boolean are like this. **Text (string)** is built from a
sequence of characters, so in most languages it isn't counted as "primitive"; it's like a
wall built from primitive bricks. You don't need to memorise these English names (*integer*,
*float*, *char*, *boolean*, *string*) now; they'll keep coming up in real code, so let them
feel familiar. For now, the only thing to keep in mind: **every piece of information has a
type, and the type decides what you can do with it.**
</Callout>

## The unchanging ones: constants

Sometimes we want a value to **never change.** A circle's pi is always 3.14159; a VAT rate
stays constant throughout a calculation. Boxes we set once and never change are called
**constants**:

```text title="Constants" showLineNumbers=false
PI       ← 3.14159
VAT_RATE ← 0.20
```

It's a convention to write constants in **CAPITAL letters**, so anyone looking at the code
immediately knows "don't touch this, it doesn't change". Making a value that shouldn't change
into a constant stops you from breaking it by accident.

## All together: a small example

Let's combine what we've learned one by one. We'll calculate the **VAT-inclusive total** of
a shopping cart. Here assignment, operations, numbers and a constant all work together:

```text title="Cart total — pseudocode" showLineNumbers=false
VAT_RATE ← 0.20

price    ← 50
quantity ← 3
subtotal ← price * quantity      # 150
vat      ← subtotal * VAT_RATE   # 30
total    ← subtotal + vat         # 180

PRINT "Amount due: " + total
```

Read it line by line: `subtotal` is computed from price and quantity, `vat` from the subtotal
and the constant rate, `total` from the sum of the two. Each variable builds on the previous
one — just like the steps in a recipe. That's how variables work together in an algorithm.

## A good variable deserves a good name

You can give a variable any name you like, but a **good** name tells you at a glance what's
inside the box.

```text title="Which is clearer?" showLineNumbers=false
# Bad                   # Good
x ← 25                  age ← 25
a ← "Ada"               name ← "Ada"
t ← f * m               total ← price * quantity
```

When you read the lines on the right you understand what they do; on the left you have to
guess. A few practical rules:

<Steps>

1. **Be meaningful** — the name should describe the value (`temperature`, `userName`).
2. **No spaces** — most languages don't accept them. Join the words: `userName` (this is
   called *camelCase*) or `user_name` (this is called *snake_case*).
3. **Don't start with a digit** — not `2player`, but `player2`.
4. **Be consistent** — use camelCase everywhere or snake_case everywhere in a project, don't
   mix them.

</Steps>

<Callout type="warning" title="Upper/lower case matters">
For most programming languages, `age`, `Age` and `AGE` are **three different boxes.** If you
create a variable as `userName` and then try to read it as `UserName`, the computer will say
"I don't know such a box". So write a name the same way you first wrote it, every time.
</Callout>

Remember: the person reading your code six months later is usually **you** — and you'll want
to thank yourself then.

<Callout type="note" title="A little history note: where did 'x' come from?">
The idea of representing something with a letter comes from **algebra**; in maths we denote
the unknown with `x`, `y`. The algebra founded by **al-Khwārizmī** — whom we met in the
[previous post](/en/blog/pseudocode) — is the ancestor of this very idea of "representing a
value with a name". The variable in programming is a descendant of the same idea: attach a
name to a value and think with it. The only difference is that in code we now prefer talking
names like `age` over single-letter `x`.
</Callout>

## Common mistakes

<Callout type="caution" title="Watch out for these traps">
- **Confusing assignment with equality:** `score ← score + 5` is not an equation, it's a
  command. Read it as "take the old value, process it, put it back".
- **Writing over a value without thinking:** Putting a new value into a box erases the old
  one. If you need to keep it, save it in a temporary variable first.
- **Forgetting/confusing quotes:** `5` is a number, `"5"` is text. They behave differently;
  numbers are calculated, texts are joined.
- **Reading a variable before giving it a value:** Trying to read a box you never put
  anything into (`PRINT total` but `total` was never defined) is a typical error.
- **Meaningless names:** Names like `x`, `y`, `tmp2` will mean nothing to you a week later.
  Let the name describe what's inside.
</Callout>

## Summary

<Callout type="tip" title="Keep in your pocket">
- A **variable** is a box in the computer's memory that has a name and stores a value. We
  represent an entity (a person, a product…) with several variables.
- **Assignment** (`←`) puts a value into a box; in code `=` means "put inside", not "is it
  equal?". Assigning one variable to another **copies** the value.
- You do **operations** with variables: arithmetic on numbers (`+ - * /`), concatenation on
  text (`+`).
- A variable's value **can change**; writing over it erases the old one. Use a **trace
  table** to follow values step by step.
- The basic primitive types: **integer**, **float**, **char**, **boolean** (and **string**,
  made of characters). Use a **constant** for values that don't change.
- Choose **meaningful names** (`age`, not `x`); use camelCase or snake_case instead of
  spaces, and be consistent about upper/lower case.
</Callout>
